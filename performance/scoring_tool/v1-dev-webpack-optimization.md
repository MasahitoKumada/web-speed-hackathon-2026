## VRT 結果

- 49 passed, 3 flaky (全てリトライで成功)
- flaky は DM 関連の pressSequentially タイムアウト (main.js 72MB のロード遅延が原因)
- 機能的な問題なし

## 考察

- webpack の production 最適化により main.js は 108MB → 72MB に削減 (-33%)
- しかし WASM バイナリ (ffmpeg-core ~30MB, magick.wasm) が `?binary` で asset/bytes としてバンドルにインライン埋め込みされたままのため、削減が限定的
- 72MB では FCP/LCP/SI にスコアがつかず、300点の閾値を超えられない
- fly.io 環境ではローカルよりもスコアが低い (サーバースペック差)

## 次のステップ

- v2: WASM バイナリのバンドル分離 + Backend gzip/brotli 圧縮
- WASM 分離で main.js を数MB まで削減 → FCP/LCP/SI にスコア発生 → 300点超え見込み
